## Что сделано

Пользователь сообщил: клавиатурное слежение опять не работает в QQ, иногда в Telegram. Мышиное слежение работает.

### Диагностика

1. Проверен live-процесс PID 31708: запущен 4-13 build из `release/patched/眼眸.exe`
2. Анализ debug.log: 5463× `Win32: no caret` + 4644× `console skip` + 941× UIA, включая 136× `BoundingRect too large (974x804)`
3. Классы окон проверены: QQ = `Chrome_WidgetWin_1`, TG = `Qt51518QWindowIcon` — `IsConsoleWindow` не даёт ложных срабатываний
4. Тест в Notepad: слежение работает → Win32 путь здоров, проблема локализована в UIA пути
5. Построена диагностическая сборка v3 с логами на `OnKeyPressed`, `TryGetCaretViaUIAutomation` и 3 пути успеха
6. По логам v3 обнаружено ключевое: в QQ `FocusedElement` **не имеет TextPattern** → фолбэк на метод B → `BoundingRect` возвращает 1060×797 (весь WebView) → псевдо-успех с позицией `(683,633)` → магнифайер "прилипает" к одной точке при каждом нажатии → пользователь воспринимает как "не следует"
7. Первичная гипотеза "Electron caret недоступен" опровергнута пользователем: штатный Windows Magnifier следит за QQ. Значит, API есть, мы его не используем
8. Пересмотр: штатный магнифайер использует **MSAA `OBJID_CARET`** через `AccessibleObjectFromWindow` (oleacc.dll) — именно этот путь у нас отсутствовал

### Кодовые изменения (TrackingManager.cs, +135/-24)

1. **Новый MSAA путь** — `AccessibleObjectFromWindow(hwnd, OBJID_CARET, IID_IAccessible)` через oleacc.dll. `accLocation` вызывается через IDispatch late binding (без зависимости от Accessibility.dll). Вставлен между Win32 и UIA как второй выбор
2. **Ужесточение метода B UIA** — `IsElementNearlyFullScreen` (порог 95%) → `LooksLikeInputControl` (width < 60% AND height < 30%). Контейнеры WebView отныне отклоняются, псевдо-успех устранён
3. **`LogThrottled` по key-buckets** — `ConcurrentDictionary<key, ticks>` вместо общего `_lastUiaFailLogTicks`. Win32-логи больше не поглощают UIA-логи (ключевой баг, скрывавший настоящие причины)
4. **Диагностические логи** — `kb_hit` в OnKeyPressed, `uia_enter` в UIA, `uia_ok_sel/vis/bnd`, `msaa_ok/fail/zero/err`

### Итерации сборки

- v2 (простое ослабление порога 0.5 → 0.95) → не решило: псевдо-успех с `(683,633)` остался
- v3 (диагностические логи) → раскрыл, что BoundingRect даёт центр контейнера
- v4 (MSAA + LooksLikeInputControl + key-buckets) → работает. Лог показывает `MSAA: OK @1037,887 size=1x18` и смещение x при печатании (1037 → 1113)

## Ключевые решения

1. **MSAA `OBJID_CARET` — канал каретки для Chromium/Electron** — штатный магнифайер использует именно этот путь. `accLocation` вызываем через late binding, чтобы избежать зависимости от Accessibility.dll
2. **AND-судья `LooksLikeInputControl`** — `width < 60% AND height < 30%`. OR-версия слишком жёстко отбраковывает широкие однострочные поля. AND точно отсекает контейнеры WebView
3. **Прямой цикл исправления, без полноценного openspec change** — пользователь просил быстрый цикл (как 4-13). Архивация change-ей при финальной уборке
4. **Фиксируем диагностические логи в production** — `kb_hit`, `uia_enter`, `msaa_ok`, `uia_ok_*`. Небольшой прирост I/O компенсируется бесценной возможностью диагностики
