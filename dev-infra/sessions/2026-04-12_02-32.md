## 完成的工作

### 诊断阶段（/opsx:explore）

用户报告"键盘跟随功能很不稳定, 经常失效"。

多假设逐一证伪：
- **ThreadPool 饥饿** — 进程已运行 57 小时, 线程数稳定在 34, 无扩容证据 → 排除
- **UIA 跨进程 RPC 永久阻塞** — `LpcReply` 线程状态从 1 自动恢复到 0, 说明卡住的 RPC 最终都会回来 → 排除
- **Hook 被 Windows `LowLevelHooksTimeout` 踢出** — 用户从 QQ 切回微信立即恢复跟随, 说明 Hook 还在回调链 → 排除
- **UIA 单应用特定问题** — 用户纠正"坏的时候所有应用都不跟随", 重新拉回到全局 latching

用 PowerShell + `System.Windows.Automation.AutomationElement.FocusedElement` 写实时探针, 复现 `TrackingManager` 同款 UIA 查询:
- 意外实证 Windows Terminal `TermControl` 面积 723290, 被 line 332 的 `> 40000` 硬编码过滤
- 光标空闲时 `GetSelection()` 返回空数组, 代码 fall through 到 BoundingRectangle 又被 40000 过滤

**根因锁定**: `TrackingManager.TryGetCaretViaUIAutomation` 的 Circuit Breaker `_consecutiveUiaTimeouts` 在 10 秒降级窗口过期后**不归零**, 导致"首次重试失败立即再合闸"死循环, 键盘跟随对所有走 UIA 路径的应用全局锁死, 只有一次 Win32 caret 成功（用户切到记事本打字）才能解锁。

**现场验证**: 用户在微信失效时切到记事本打字一次, 然后回到微信立刻恢复跟随。完美匹配"Win32 成功归零 counter"路径。

另外确认的副作用: `TrackingManager` 中所有诊断路径均使用 `System.Diagnostics.Debug.WriteLine`, 在 Release 编译下被 `[Conditional("DEBUG")]` 剪掉, `%APPDATA%\WindowsMagnifier\debug.log` 与 `error.log` 里**零 Tracking 记录**, 生产零可观测性 —— 这也是用户和开发者都一直"感觉有问题但无法自证"的原因。

- **P0 核心修复**: `TryGetCaretViaUIAutomation` 入口重构为三段式:
  ```csharp
  var disabledUntil = Interlocked.Read(ref _uiaDisabledUntilTicks);
  if (disabledUntil > 0)
  {
      if (DateTime.UtcNow.Ticks < disabledUntil)
          return false;
      Interlocked.Exchange(ref _uiaDisabledUntilTicks, 0);
      Interlocked.Exchange(ref _consecutiveUiaTimeouts, 0);
      _log.LogDebug("Tracking", "UIA backoff expired, counter reset");
  }
  ```
  归零点紧贴降级判定的唯一检查点, 未来新增路径会自动继承正确行为。
